# Plan de implementación — MVP

> **Para workers agénticos:** cada fase trae **prompt, modelo, esfuerzo y skills**. Ejecutar fase por fase con checkpoint de revisión entre fases. Steps de código reales se generan dentro de cada fase con la skill `test-driven-development`.

**Objetivo:** app vendible mínima — un usuario se registra y verifica identidad (1), compra una bebida a escrow (3), invita a un amigo (4) y completa una cita verificada con liberación de pago (5).

**Arquitectura:** React Native + Expo (cliente nativo) sobre Supabase (Auth, Postgres+RLS, Realtime, Storage, Edge Functions). Toda lógica de dinero/seguridad vive en Edge Functions con `service_role`; el cliente nunca calcula ni mueve saldo. Ledger append-only + idempotencia desde el día 1.

**Sub-proyectos incluidos:** 1 (Identidad), 3 (Tienda+Bar+Wallet/Escrow), 4 (Invitaciones+Chat), 5 (Motor de cita). Match básico (swipe sin gating) se incluye ligero dentro de la fase 4.x de descubrimiento simple.

**Referencia de diseño:** `docs/2026-06-30... ` → ver `2026-07-01-modelo-negocio-design.md` secciones A, D, E, F, G.

**Regla de oro por fase:** toda fase que toca dinero, escrow, QR, geofence o RLS → revisión de seguridad obligatoria antes de merge (`security-review` + `requesting-code-review`).

**Decisiones cerradas:**
1. **Partner de escrow: Red Pontis** (subcuentas/custodia). Se integra detrás de una interfaz `PaymentProvider`. Afecta fases 3.1 y 5.4.
2. **Suscripción premium vía IAP** de Apple/Google (~15% comisión; fuera de MVP, sub-proyecto 8). Las **bebidas** NO van por IAP (servicio del mundo real) → Red Pontis.

# Sub-proyecto 1 — Identidad y perfiles

**Meta:** registro + KYC + perfiles amigo/rentador con las "costuras" (feature flags, columnas de nivel, tabla de suscripción) listas aunque el MVP no las use aún.

### Fase 1.0 — Scaffold del proyecto

- **Objetivo:** monorepo Expo + Supabase, estructura de carpetas, env, linting, CI básico, git init.
- **Prompt para Claude:**
  > Inicializa un proyecto nuevo React Native con Expo (TypeScript) llamado `amigos-app` y un proyecto Supabase asociado. Crea la estructura: `app/` (rutas Expo Router), `components/`, `lib/` (cliente supabase, helpers), `supabase/` (migrations, functions), `tests/`. Configura ESLint + Prettier + TypeScript estricto. Añade `.env.example` con `SUPABASE_URL`, `SUPABASE_ANON_KEY`, `SUPABASE_SERVICE_ROLE_KEY` (solo Edge Functions), `TRUORA_API_KEY`. Configura un workflow CI (lint + typecheck + test). Haz `git init` y primer commit. No implementes features todavía.
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** using-git-worktrees, executing-plans

### Fase 1.1 — Esquema de datos + RLS + storage (CRÍTICO)

- **Objetivo:** tablas base con RLS estricto y las costuras de expansión.
- **Prompt para Claude:**
  > Con TDD (tests de RLS con `supabase test db`), crea la migración inicial. Tablas: `profiles` (id=auth.uid, rol enum `amigo`|`rentador`, nombre, alias, edad, genero, profesion, foto_url, hobbies text[], intereses text[], `kyc_estado` enum `pendiente`|`verificado`|`rechazado`, `verificado_at`); `preferencias_salida` (perfil_id, sexo_pref, tipo_salida, rango_edad int4range, horario, distritos text[]) — solo se usará para gating en premium pero se crea ya; columnas de nivel en `profiles`: `recaudacion_acumulada numeric default 0`, `gasto_acumulado numeric default 0`, `nivel text default 'bronce'`, `nivel_actualizado_at`; tabla `suscripcion` (perfil_id, tipo, estado enum `activa`|`inactiva`, vigente_hasta) — vacía en MVP; tabla `feature_flags` o columna `flags jsonb default '{}'`. Enums de estado de cita completos aunque MVP no los use todos: `pendiente`,`confirmada`,`en_curso`,`finalizada`,`no_show`,`disputa`. RLS: cada usuario ve/edita solo su fila; nadie lee el DNI de otro; saldos nunca expuestos a otro usuario. Crea bucket privado `dni` (no público) y bucket `fotos` con políticas por owner. Escribe tests que prueben que un usuario NO puede leer el perfil KYC ni el DNI de otro.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 1.2 — Auth (OTP teléfono + email)

- **Objetivo:** login/registro.
- **Prompt para Claude:**
  > Con TDD, implementa el flujo de auth con Supabase Auth: registro/login por teléfono (OTP SMS) y email. Pantallas Expo Router: `sign-in`, `verify-otp`, selección de rol (`amigo`/`rentador`) en primer login que crea la fila `profiles`. Maneja errores (OTP inválido, expirado, reenvío con cooldown). Estado de sesión persistente.
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** frontend-design, test-driven-development

### Fase 1.3 — Alta de perfil (amigo / rentador)

- **Objetivo:** formularios de perfil completos.
- **Prompt para Claude:**
  > Con TDD para la validación, implementa el onboarding de perfil. Campos amigo Y rentador: nombre, alias, edad (≥18, validado), género, profesión, intereses, hobbies, foto. El amigo además captura `preferencias_salida` (aunque en MVP no filtre). Guarda en `profiles`/`preferencias_salida`. Validación client + server (Edge Function o RLS check). Bloquea avanzar sin campos obligatorios.
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** frontend-design, test-driven-development

### Fase 1.4 — KYC Truora (DNI + liveness) (CRÍTICO)

- **Objetivo:** verificación de identidad de AMBOS roles.
- **Prompt para Claude:**
  > Con TDD, integra Truora KYC. Cliente: captura DNI (cámara nativa) + selfie con liveness, sube el DNI al bucket privado `dni`. Edge Function `kyc-start` crea la verificación en Truora (doc + liveness + cruce RENIEC) usando `TRUORA_API_KEY` (server-side, nunca en cliente). Edge Function `kyc-webhook` recibe el resultado, valida firma del webhook, y actualiza `profiles.kyc_estado`=`verificado`/`rechazado` + `verificado_at`. Idempotente (mismo webhook dos veces no duplica). Gate: un usuario sin `kyc_estado=verificado` no puede comprar ni recibir dinero. Tests: webhook firma inválida rechazado; doble webhook idempotente; usuario no verificado bloqueado.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review, systematic-debugging

### Fase 1.5 — Ver/editar perfil + foto

### Fase 1.6 — Endurecimiento y revisión

- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** requesting-code-review, security-review, verification-before-completion

# Sub-proyecto 3 — Tienda + Bar + Wallet/Escrow

**Meta:** comprar bebidas → fondos a escrow → stock en el "bar". Núcleo de dinero. Depende de sub-proyecto 1 (usuario verificado).

### Fase 3.0 — Ledger + wallet + escrow (schema) (CRÍTICO)

- **Objetivo:** contabilidad append-only e idempotente.
- **Prompt para Claude:**
  > Con TDD, crea el modelo de dinero. Tablas: `ledger` append-only (id, perfil_id, tipo enum `compra`|`escrow_lock`|`escrow_release`|`payout`|`refund`|`fee`, monto numeric, moneda, referencia_id, `idempotency_key` unique, created_at) — sin UPDATE/DELETE (revoca esos permisos por RLS/trigger); `bebidas_catalogo` (id, nombre, tipo_invitacion enum `divertida`|`romantica`|`misteriosa`|`amigos`|`autor`, valor_v numeric, activo bool); `bar` (perfil_id rentador, bebida_id, estado enum `disponible`|`bloqueada`|`consumida`, escrow_ref); vista `balance` calculada desde el ledger (nunca un número mutable suelto). RLS: cada quien ve solo su ledger/bar/balance. Trigger que impida modificar filas de ledger. Tests: balance = suma correcta del ledger; no se puede UPDATE una fila de ledger; idempotency_key duplicada rechazada.
- **Modelo:** Opus 4.8
- **Esfuerzo:** xhigh
- **Skills:** test-driven-development, security-review

### Fase 3.1 — Integración pagos + escrow (CRÍTICO)

- **Objetivo:** cobro real y custodia en partner.
- **Prompt para Claude:**
  > Con TDD (mock del partner en tests), integra **Red Pontis** (escrow/custodia con subcuentas) detrás de una interfaz `PaymentProvider` para poder cambiarlo. Edge Function `comprar-bebida`: recibe bebida_id, calcula el buyer fee server-side según tier del rentador (MVP: todos free = +15%) + costo de procesamiento, crea la orden de pago, al confirmarse mueve los fondos a escrow (subcuenta del partner), escribe al `ledger` (`compra`, `fee`, `escrow_lock`) con `idempotency_key`, y añade la bebida al `bar` como `disponible`. Webhook de confirmación de pago idempotente. Tests: fee calculado en server (cliente no puede alterarlo); doble webhook no duplica bebida; fallo de pago no crea stock.
- **Modelo:** Opus 4.8
- **Esfuerzo:** xhigh
- **Skills:** test-driven-development, systematic-debugging, security-review

### Fase 3.2 — Catálogo + Tienda (UI)

- **Objetivo:** ver y comprar bebidas.
- **Prompt para Claude:**
  > Con TDD ligero, pantalla Tienda que lista `bebidas_catalogo` activas con V y tipo de invitación, y dispara `comprar-bebida`. Muestra desglose transparente: V + buyer fee + procesamiento = total. Estados de carga/éxito/error. El catálogo es config del operador (seed inicial con las 5 bebidas de ejemplo del diseño).
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** frontend-design, test-driven-development

### Fase 3.3 — Bar / stock (UI)

- **Objetivo:** ver stock disponible para invitar.
- **Prompt para Claude:**
  > Con TDD ligero, pantalla Bar del rentador: lista sus bebidas en `bar` con estado (`disponible`/`bloqueada`/`consumida`). Cada disponible tiene acción "Invitar" (que consumirá el sub-proyecto 4). Muestra el balance calculado del ledger.
- **Modelo:** Sonnet 5
- **Esfuerzo:** low
- **Skills:** frontend-design, test-driven-development

### Fase 3.4 — Conciliación y revisión

- **Objetivo:** cerrar dinero seguro.
- **Prompt para Claude:**
  > Revisa el sub-proyecto 3: verifica idempotencia en todos los caminos de pago/escrow, que ningún cálculo de fee ocurra en cliente, integridad del ledger, y conciliación balance↔partner. Añade un job/función de conciliación que detecte discrepancias. Corre la suite completa y confirma verde.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** requesting-code-review, security-review, verification-before-completion

# Sub-proyecto 4 — Invitaciones, solicitudes, chat y confirmación

**Meta:** rentador invita (desde el bar) o amigo solicita; se acepta; se abre chat; se confirma la cita. Depende de 1 y 3. Incluye descubrimiento simple (swipe básico sin gating).

### Fase 4.0 — Schema de invitaciones + chat

- **Objetivo:** modelo de datos.
- **Prompt para Claude:**
  > Con TDD, crea tablas: `invitaciones` (id, emisor_id, receptor_id nullable si global, tipo enum `invitacion`(rentador→amigo)|`solicitud`(amigo→rentador), alcance enum `especifica`|`global`, bebida_bar_id, tiempo_estimado_min int, zona_aproximada, estado enum `pendiente`|`aceptada`|`rechazada`|`expirada`, created_at); `citas` (id, invitacion_id, estado usando el enum de cita completo, zona, hora, mensaje, confirmada_at); `chat_mensajes` (cita_id, emisor_id, texto, created_at). RLS: solo las dos partes ven su invitación/cita/chat. Realtime habilitado en `chat_mensajes` y `citas`. Tests RLS de aislamiento.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 4.1 — Descubrimiento simple (swipe básico)

- **Objetivo:** ver perfiles uno por uno (sin gating, MVP).
- **Prompt para Claude:**
  > Con TDD ligero, vista tipo Bumble (un perfil a la vez, aceptar/pasar) que lista perfiles verificados del rol opuesto. Sin filtros premium ni gating por nivel (eso es sub-proyecto 2). Desde un perfil, el rentador puede iniciar una invitación con una bebida de su bar; el amigo puede iniciar una solicitud.
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** frontend-design, test-driven-development

### Fase 4.2 — Crear invitación / solicitud + bloqueo de fondos

- **Objetivo:** emitir propuesta y bloquear la bebida.
- **Prompt para Claude:**
  > Con TDD, Edge Function `crear-invitacion`: valida que el emisor esté verificado; si es invitación de rentador, valida que la bebida del bar esté `disponible` y la pasa a `bloqueada` (los fondos ya están en escrow desde la compra); guarda la fila `invitaciones` con bebida, tiempo_estimado y zona. Para solicitud de amigo, no hay bebida aún (la pondrá el rentador al aceptar). Idempotente. Tests: no se puede invitar con bebida ya bloqueada; emisor no verificado rechazado.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 4.3 — Aceptar / rechazar → abre chat

- **Objetivo:** transición a chat.
- **Prompt para Claude:**
  > Con TDD, Edge Function `responder-invitacion`: aceptar crea la `cita` en estado `confirmada`-previo (aún sin detalles) y habilita el chat; rechazar libera la bebida del bar (`bloqueada`→`disponible`). Si una solicitud de amigo es aceptada por un rentador, éste asigna una bebida de su bar (se bloquea). Notifica push a la contraparte. Tests: rechazo devuelve la bebida a disponible; aceptar habilita chat solo a las dos partes.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 4.4 — Chat realtime + moderación anti-fuga

- **Objetivo:** chat con detección de pago por fuera.
- **Prompt para Claude:**
  > Con TDD, chat en tiempo real (Supabase Realtime) entre las dos partes. Edge Function o trigger de moderación que marca/oculta mensajes con patrones de número de teléfono / CBVU / intento de pago externo y emite warning; reincidencia registra flag en el perfil. Tests: mensaje con teléfono es marcado; usuarios ajenos no acceden al chat.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 4.5 — Confirmar cita (resumen)

- **Objetivo:** botón "confirmar cita".
- **Prompt para Claude:**
  > Con TDD, en el chat el amigo tiene "Confirmar cita": captura zona de encuentro, hora y mensaje adicional; al confirmar, la `cita` pasa a `confirmada` con esos detalles y se muestra a ambos un resumen "Cita confirmada" (bebida, V, tiempo estimado, zona, hora). Notifica push a ambos. Tests: solo el amigo puede confirmar; confirmar setea estado y detalles.
- **Modelo:** Sonnet 5
- **Esfuerzo:** medium
- **Skills:** frontend-design, test-driven-development

### Fase 4.6 — Revisión

- **Prompt para Claude:**
  > Revisa el sub-proyecto 4: máquina de estados de invitación/cita coherente, aislamiento RLS de chat, moderación funcionando, y que el bloqueo/liberación de bebida sea consistente con el escrow. Corre la suite y confirma verde.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** requesting-code-review, security-review, verification-before-completion

# Sub-proyecto 5 — Motor de cita (QR, geofence, cronómetro, extensión, payout)

**Meta:** verificar el encuentro y liberar el pago. El corazón de confianza + dinero. Depende de 3 y 4.

### Fase 5.0 — Sesión + token QR rotativo (CRÍTICO)

- **Objetivo:** QR anti-screenshot.
- **Prompt para Claude:**
  > Con TDD, Edge Function `cita-session`: genera para una cita `confirmada` un token dinámico que rota cada ~30s (HMAC server-side de `cita_id` + ventana de tiempo + secreto). El cliente pide el token vigente y lo renderiza como QR; el otro lo escanea y el backend valida contra la ventana válida. Nunca es estático ni predecible desde el cliente. Tests: token expira tras la ventana; token de otra cita no valida; cliente no puede fabricar un token válido.
- **Modelo:** Opus 4.8
- **Esfuerzo:** xhigh
- **Skills:** test-driven-development, security-review

### Fase 5.1 — Scan mutuo + geofence (CRÍTICO, activo desde MVP)

- **Objetivo:** verificación presencial real.
- **Prompt para Claude:**
  > Con TDD, Edge Function `cita-verificar-inicio`: recibe el token escaneado + coords GPS de ambos dispositivos; valida token vigente Y que ambos estén dentro de ~50–100 m (geofence, distancia haversine server-side); requiere confirmación bidireccional (ambos escanean/confirman). Al validar, pasa la cita a `en_curso` y setea `inicio_at`. Tests: fuera del radio rechazado; token expirado rechazado; un solo lado no inicia la cita.
- **Modelo:** Opus 4.8
- **Esfuerzo:** xhigh
- **Skills:** test-driven-development, security-review, systematic-debugging

### Fase 5.2 — Cronómetro + notificaciones

- **Objetivo:** contar el tiempo y avisar.
- **Prompt para Claude:**
  > Con TDD, cronómetro de cita basado en `inicio_at` + `tiempo_estimado_min`. Programa (Edge Function + Supabase cron o `pg_cron`) notificaciones push (Expo Push): una a 15 min del fin y otra al fin. UI en curso muestra el tiempo restante en vivo. Tests: cálculo de fin correcto; notificación de 15 min se agenda; extensión reprograma (ver 5.3).
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development

### Fase 5.3 — Extensión de cita (bebida adicional)

- **Objetivo:** alargar la cita en curso.
- **Prompt para Claude:**
  > Con TDD, Edge Function `cita-extender`: durante `en_curso`, el rentador invita una bebida adicional de su bar (se bloquea, fondos ya en escrow); el amigo acepta o rechaza (push). Si acepta, suma el `tiempo_estimado` de esa bebida al cronómetro y reprograma las notificaciones. Escribe al ledger el nuevo `escrow_lock`. Tests: rechazo no extiende ni bloquea; aceptar extiende el fin y reprograma avisos; bebida no disponible rechazada.
- **Modelo:** Opus 4.8
- **Esfuerzo:** high
- **Skills:** test-driven-development, security-review

### Fase 5.4 — Liberación de pago + no-show (CRÍTICO)

- **Objetivo:** mover escrow → balance del amigo, o refund.
- **Prompt para Claude:**
  > Con TDD, Edge Function `cita-liquidar` disparada al fin del cronómetro (sin scan de fin — decisión de diseño): si la cita llegó a `en_curso` y no hay disputa/pánico, libera el escrow del partner y escribe al ledger `escrow_release` + `payout` al amigo aplicando el seller fee correcto (MVP: free = −20%) neto de procesamiento; pasa la cita a `finalizada` y la bebida a `consumida`; suma a `recaudacion_acumulada`/`gasto_acumulado` (para nivel, aunque el gating sea sub-proyecto 6). **No-show:** si nunca hubo scan de inicio dentro de la ventana (~30 min post-hora), pasa a `no_show`, hace refund del escrow al bar del rentador (bebida `disponible` de nuevo) y NO cobra al amigo. Todo idempotente. Tests: payout aplica fee correcto; doble liquidación no duplica; no-show refund correcto; disputa bloquea liquidación automática.
- **Modelo:** Opus 4.8
- **Esfuerzo:** xhigh
- **Skills:** test-driven-development, security-review

### Fase 5.5 — E2E del flujo completo + revisión final

- **Objetivo:** probar la cita punta a punta y cerrar el MVP.
- **Prompt para Claude:**
  > Escribe un test end-to-end del flujo MVP completo: registro+KYC (mock) → compra de bebida → invitación → aceptación → chat → confirmar cita → scan+geofence inicio → cronómetro → fin → payout. Verifica el ledger cuadrado en cada paso. Audita seguridad de QR, geofence y liberación de fondos. Corre toda la suite y confirma verde con evidencia.
- **Modelo:** Opus 4.8
- **Esfuerzo:** max
- **Skills:** requesting-code-review, security-review, verification-before-completion
